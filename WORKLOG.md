## 1. 프로젝트 한눈에
- 주제: **한식 메뉴판 인식(CV/OCR) → LLM 다국어(영어 우선) 해석·가이드.**
- 흐름: 사진 1장 → PaddleOCR(한국어) 인식 → LLM이 소개·맛·재료·알레르기 주의를 목표 언어로 설명.
- 규칙: 추천/필터링 없음, 헬스케어 아님, 알레르기 비단정('의심 재료'만 + "식당에 직접 문의"), 가격은 결과 미포함.
- 대상: 방한 외국인 관광객(K-관광).

## 2. 방향 전환 배경 (왜 바꿨나 — 핵심 판단)
- v1(유럽→한국어)에서 정확도 병목이 **OCR 단계**로 확인됨: 프·독 특수문자(é·ü·ç), 장식체, 저대비. PaddleOCR·Tesseract 둘 다 그 언어에 완전 최적화는 아님.
- **뒤집으면 병목이 풀린다**: 한국어 OCR은 데이터·수요가 많아 성숙(PaddleOCR 한국어 모델 양호). 게다가 원어민이라 **정답 라벨·검증이 쉬움**(v1에서 raifort→roquefort 오인식을 사람이 놓칠 뻔).
- 트레이드오프: 해석 방향이 한국어→외국어라 **번역 검증 부담**이 새로 생김. 데이터도 새로(한식 메뉴판) 수집 필요. 단, 한국이라 촬영이 쉬움.

## 3. OCR 전략 (핵심)
- **주력 = PaddleOCR(한국어).** 베이스라인 → 개선(전처리 등) → 오류사례 분석의 모델링 스토리.
- **고성능 비교 기준 = CLOVA OCR.** 상용 고정확 OCR을 벤치마크로, PaddleOCR과 정량 비교 → 개선 여지·한계 제시.
- v1 검증 방법론(CLAHE 전처리, compare_menu.py 채점) 언어만 바꿔 재사용.

## 6-1. LLM 파이프라인 구축 & OCR 개선 (2026-07-02, 이어서)
- `menu_to_prompt.py` **한국어 적응 완료**: looks_like_name 한글 허용(2글자+), 콘솔 UTF-8 고정, 기본 --lang korean/--target English, 프롬프트 한식 프레이밍(독일어 규칙 제거).
- **LLM 해석 파이프라인 end-to-end 동작 확인**(OpenAI gpt-4o-mini/gpt-4o): OCR → 필터 → LLM → `explained.json`.
- **프롬프트 재작성(환각 억제)**: "이 식당 실제 레시피 모름 → 일반 상식만 typically/often로 헤징", 없는 재료(된장 등) 단정 금지, 'earthy/흙내' 등 모호·부정 풍미어 금지. + **왕초보 배려**: 요리유형어(국수·전골·사리…)·장류(된장·고추장·간장·청국장)가 처음 나오면 목표언어로 필수 풀이. + 특정 이미지가 아니라 어떤 메뉴판에도 적용되게 범용화.
- **OCR 글자붙음 병합 추가**(`merge_line_fragments`): 메뉴판 디자인 띄어쓰기로 쪼개진 한글 조각(공/기/밥→공기밥)을 좌표(같은 줄+가로 근접)로 병합. 숫자(가격) 박스는 병합 장벽 → 이름·가격 오병합(뼈전골*38000) 방지.
- **OCR 모델 PP-OCRv3 → PP-OCRv5 전환(핵심)**: 6.jpg(225px 저해상도) 인식 0/4 → **4/4**. ★ 실험으로 **CLAHE·업스케일 전처리는 불필요하고 v5엔 오히려 해로움** 확인 → v1의 CLAHE 방법론은 한국어 v5에 안 맞아 폐기.
- **compare_menu.py 한국어 버그 수정**: norm()이 한글을 전부 지워 모든 메뉴가 유사도 1.0으로 오매칭되던 것 → NFC 재결합 + `가-힣` 유지로 수정(그 전엔 "언어 무관"이 사실과 달랐음).
- **4.jpg 채점 개선 궤적**(원어민 truth 8개 대비): 3/8 → (병합)6/8 → (v5)**일치 7/8·누락 0**. 잔여 잡티 = 간판 제목("얼큰한 해장국")을 요리로 오생성 1건.

## 6-2. "왜 파인튜닝(직접 학습)으로 안 올렸나" (발표용 판단 — 정직한 한계)
- **정확도 향상의 실제 지렛대는 '우리가 학습'이 아니라 ① 더 강한 사전훈련 모델 선택(PP-OCRv3→v5) + ② 후처리(좌표 병합·LLM 철자 교정)였다.** (측정: 6.jpg 50→100%, 4.jpg 3/8→7/8)
- 직접 rec 파인튜닝을 안/못 한 이유:
  1. **데이터 부족** — rec 파인튜닝은 단어 crop+라벨 수천~수만 장 필요. 보유 = 메뉴판 6장 → 과적합, 대규모 한국어로 사전학습된 v5를 이길 수 없음.
  2. **자원 한계** — 로컬 GPU 없음(CPU 학습 비현실적), 파인튜닝 노트북은 Colab T4 전용.
  3. **오류의 성격** — 남은 실패의 다수가 '인식'이 아니라 '구조'(세로쓰기·글자 사이 띄어쓰기) → 학습으로 못 고치고 좌표 후처리가 정답.
- 결론: 이 문제의 정석은 "OCR을 새로 학습"이 아니라 **"강한 사전모델 + 강한 후처리/LLM 이해층"**. 파인튜닝은 데이터가 대폭 늘어야 의미 있는 '향후 과제'로 남김. (부정적 결과도 정직한 실험 설계로 발표에 사용)
